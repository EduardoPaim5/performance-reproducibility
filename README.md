# Performance Reproducibility

Initial repository for an academic experiment on the reproducibility of performance tests for web servers. The HTTP target is a static page served by Nginx, and load is generated by k6 from a dedicated local client.

## Objective

Compare the performance behavior of four ways to run Nginx:

- C1: native Nginx on a local server.
- C2: Nginx in local Docker.
- C3: Nginx in local Docker Swarm.
- C4: Nginx in Docker Swarm on an AWS EC2 t3.micro instance.

Initial monitoring uses Prometheus, Node Exporter, and cAdvisor. Grafana is provisioned in the local environment for later data inspection and visualization.

## Article alignment

The current article version defines:

- 60-second ramp-up;
- 1-minute warm-up;
- 5-minute stable collection;
- 30 independent runs per load level and per scenario;
- Nginx 1.28.x;
- Docker Engine 27.3 and Docker Compose 2.29 as planned versions;
- Ubuntu Server 24.04 LTS on the AWS EC2 t3.micro instance;
- coefficient of variation as the primary reproducibility metric.

The k6 script tags each request with `phase`: `ramp_up`, `warmup`, or `measurement`. To discard warm-up during statistical analysis, use `EXPORT_RAW=true` and process only samples with `phase=measurement`.

## Recommended topology

To reduce interference in the load generator, the recommended topology is:

- Notebook or dedicated client: runs only k6 and the load scripts.
- Stronger local server: runs C1, C2, C3, Prometheus, Grafana, Node Exporter, and cAdvisor.
- AWS EC2 t3.micro: runs C4, Node Exporter, and cAdvisor.

This keeps the test client the same in all scenarios, while the target changes between the local server and AWS. For C4, Prometheus on the local server collects AWS exporters through an SSH tunnel, without exposing Node Exporter or cAdvisor publicly.

## Experimental phases

The experiment should run in two phases to reduce computational cost and runtime.

### Exploratory phase

Objective: identify the load range where each scenario starts to degrade.

Use a small number of replications per level. To avoid accidental long runs, the script defaults are short and work as a local smoke test. For the real exploratory phase, explicitly use:

```bash
EXPERIMENT_PHASE=exploratory
EXPLORATORY_REPLICATIONS=3
RAMP_UP=1m
WARMUP=1m
DURATION=5m
COOLDOWN=30
```

In this phase, run the initial progressive levels and keep increasing load until degradation signals appear, such as a substantial latency increase, throughput drop, HTTP errors, or resource saturation.

### Confirmatory phase

Objective: collect statistically relevant data.

Use:

```bash
EXPERIMENT_PHASE=confirmatory
REPLICATIONS=30
```

This phase should not unnecessarily repeat all exploratory levels. Run only the levels selected after the exploratory phase, focusing collection on the points relevant to statistical comparison.

## Prerequisites

Install only what is necessary for the scenario being run.

- Bash.
- k6 on the load client.
- curl, recommended for simple checks.
- jq, optional, to read `k6/scenarios/load-levels.json`.
- Docker and Docker Compose for C2.
- Docker Swarm for C3 and C4.
- Nginx installed on the local host for C1.
- Node Exporter and cAdvisor according to the monitored scenario.
- An AWS EC2 t3.micro instance already created and manually prepared for C4.

This repository does not create AWS resources, store credentials, or change security rules.

## C1: local native Nginx

Run:

```bash
cd performance-reproducibility
./scripts/run-local-native.sh
```

By default, Nginx listens on `http://127.0.0.1:8080/`. To change the port:

```bash
PORT=8082 ./scripts/run-local-native.sh
```

The script creates an isolated configuration in `results/native-nginx/` and does not change the system Nginx service.

## C2: Nginx in local Docker

Run:

```bash
cd performance-reproducibility
./scripts/run-docker.sh
```

Main services:

- Nginx: `http://127.0.0.1:8080/`
- Prometheus: `http://127.0.0.1:9090/`
- Grafana: `http://127.0.0.1:3000/`
- cAdvisor: `http://127.0.0.1:8081/`
- Node Exporter: `http://127.0.0.1:9100/`

## C3: Nginx in local Docker Swarm

If Swarm is already active:

```bash
cd performance-reproducibility
./scripts/run-swarm-local.sh
```

If the host is not in Swarm mode yet and you want to initialize it explicitly:

```bash
INIT_SWARM=true ./scripts/run-swarm-local.sh
```

The default stack is named `perf-local`. To change it:

```bash
STACK_NAME=my-stack ./scripts/run-swarm-local.sh
```

## C4: Nginx in Docker Swarm on AWS

Before using C4, manually prepare the EC2 t3.micro instance:

- Docker installed.
- Swarm initialized.
- HTTP port `80` allowed only according to the experiment policy.
- Node Exporter and cAdvisor installed or published in a monitoring-compatible way.
- Addresses reviewed in `monitoring/prometheus/prometheus.yml`.
- Local SSH tunnel configured if local Prometheus needs to collect exporters from AWS.

The AWS script does not run anything remotely without `DEPLOY_AWS=true`.

```bash
cd performance-reproducibility
AWS_SSH_HOST=aws-ec2-public-dns.example.com ./scripts/run-aws-swarm.sh
```

To publish through SSH after reviewing the placeholders:

```bash
AWS_SSH_HOST=aws-ec2-public-dns.example.com \
AWS_SSH_USER=ec2-user \
NGINX_PUBLISHED_PORT=80 \
DEPLOY_AWS=true \
./scripts/run-aws-swarm.sh
```

Do not place keys, tokens, or credentials inside the repository.

To monitor Node Exporter and cAdvisor on AWS without exposing public ports, use the strategy documented in `docs/aws-ssh-tunnel-prometheus.md`.

## Running k6

Simple single test:

```bash
TARGET_URL=http://127.0.0.1:8080/ VUS=50 RAMP_UP=10s WARMUP=10s DURATION=1m k6 run k6/scripts/load-test.js
```

Short validation run using the defined levels:

```bash
SCENARIO=C2 TARGET_URL=http://127.0.0.1:8080/ ./scripts/run-k6-levels.sh
```

Without other variables, this command uses short defaults: 1 replication per level, `RAMP_UP=5s`, `WARMUP=5s`, `DURATION=15s`, `COOLDOWN=0`, and `EXPORT_RAW=true`.

Recommended exploratory run:

```bash
SCENARIO=C2 \
TARGET_URL=http://127.0.0.1:8080/ \
EXPERIMENT_PHASE=exploratory \
EXPLORATORY_REPLICATIONS=3 \
RAMP_UP=1m \
WARMUP=1m \
DURATION=5m \
COOLDOWN=30 \
EXPORT_RAW=true \
./scripts/run-k6-levels.sh
```

Confirmatory run aligned with the article, with 30 replications per selected level:

```bash
SCENARIO=C2 \
TARGET_URL=http://127.0.0.1:8080/ \
EXPERIMENT_PHASE=confirmatory \
REPLICATIONS=30 \
RAMP_UP=1m \
WARMUP=1m \
DURATION=5m \
EXPORT_RAW=true \
./scripts/run-k6-levels.sh
```

Initial levels are in `k6/scenarios/load-levels.json`:

- 50 VUs
- 75 VUs
- 112 VUs
- 168 VUs
- 252 VUs
- 378 VUs

After these levels, keep increasing by approximately 50% until observing saturation or clear degradation.

For the confirmatory phase, adjust `k6/scenarios/load-levels.json` or point `LEVELS_FILE` to a file containing only the levels selected after the exploratory phase.

## Initial result summarization

After running one or more runs with `--summary-export`, generate a simple CSV:

```bash
python3 analysis/summarize-k6-results.py \
  --input-dir results/raw \
  --output results/processed/k6-summary.csv
```

The CSV includes columns for scenario, phase, load, replication, mean latency, p95, p99, throughput, and error rate.

## Result collection

After a run:

```bash
SCENARIO=C2 ./scripts/collect-results.sh
```

The script saves metadata, available Docker information, and Prometheus target state in `results/`.

## Initial interpretation

Compare scenarios using equivalent metrics:

- mean latency, p90, p95, and p99 from k6;
- requests per second;
- HTTP error rate;
- CPU and memory usage from Node Exporter;
- container usage from cAdvisor;
- signs of network, disk, or CPU saturation.

Also record external conditions: hardware, versions, operating system, network, AWS region, test time, and changed settings.

## Mandatory manual review

Before running the full experiments, review:

- `monitoring/prometheus/prometheus.yml`: replace AWS placeholders and confirm ports.
- `k6/scenarios/load-levels.json`: adjust duration per level according to the article.
- `nginx/nginx.conf`: confirm that the configuration will be the same in all scenarios.
- `docker/docker-compose.yml`: confirm volume permissions for cAdvisor and Node Exporter on your system.
- `docker/swarm/stack.yml`: confirm replicas, constraints, and published port.
- `scripts/run-aws-swarm.sh`: confirm SSH user, host, remote directory, and network policies.
- `docs/aws-ssh-tunnel-prometheus.md`: confirm `AWS_USER`, `AWS_HOST`, `SSH_KEY`, and local tunnel ports.
- Actual environment versions: compare Nginx, Docker, Docker Compose, k6, and Ubuntu with the versions declared in the article.
- AWS monitoring: do not expose Node Exporter or cAdvisor publicly; prefer remote Prometheus, VPN, private network, or SSH tunnel.
- Firewall/security group rules: allow only the minimum required access.

## Security notes

- The repository does not contain credentials.
- The scripts do not remove resources or run destructive operations.
- Local Swarm initialization requires `INIT_SWARM=true`.
- Remote publication to AWS requires `DEPLOY_AWS=true`.

## Before publishing on GitHub

Files in `results/` are local run data and should not be versioned. `.gitignore` keeps only the `.gitkeep` files in subdirectories.

Recommended checklist:

```bash
find results -type f ! -name .gitkeep
git status --short --ignored
```

If any result file appears as versioned, remove it from the index before pushing. Do not publish SSH keys, AWS credentials, `.env` files, or raw datasets without review.
