# Initial experimental plan

## Research question

How do different deployment modes for the same Nginx server affect the reproducibility of HTTP performance tests under progressive load?

## Scenarios

- C1: native Nginx on a local server.
- C2: Nginx in local Docker.
- C3: Nginx in local Docker Swarm.
- C4: Nginx in Docker Swarm on AWS EC2 t3.micro.

## Controlled variables

- Same static content in `nginx/html/index.html`.
- Same base Nginx configuration whenever possible.
- Same load generation client.
- Same k6 script.
- Same initial VU levels.
- Same duration per level within a comparative run.

## Run topology

The recommended topology separates load generation and the target server:

- Test client: runs only k6 and load scripts.
- Local server: runs C1, C2, C3, and the local monitoring infrastructure.
- AWS EC2 t3.micro instance: runs C4 and exporters.

Prometheus and Grafana stay with the monitoring environment on the local server, not on the k6 client. This decision reduces the chance that Prometheus/Grafana compete with k6 on the notebook or dedicated client. The cost is that Prometheus, Grafana, and exporters introduce a small load on the local server, which must be kept constant and documented in scenarios C1, C2, and C3.

## Independent variables

- Nginx deployment mode.
- Local or AWS environment.
- Orchestration layer: no container, Docker, or Docker Swarm.

## Dependent variables

- Throughput in requests per second.
- Mean latency, p90, p95, and p99.
- HTTP error rate.
- Host CPU, memory, network, and disk.
- Container CPU and memory when applicable.

## Progressive load

Initial levels are:

| Level | VUs |
|------:|----:|
| 1 | 50 |
| 2 | 75 |
| 3 | 112 |
| 4 | 168 |
| 5 | 252 |
| 6 | 378 |

After level 6, keep increasing by approximately 50% per level until observing degradation or saturation.

Each run should follow the temporal profile described in the article:

- 1 minute of ramp-up until the target load;
- 1 minute of warm-up;
- 5 minutes of stable collection;
- discard `ramp_up` and `warmup` samples in the final statistical analysis.

The file `k6/scripts/load-test.js` records each request phase in the `phase` tag, allowing only `phase=measurement` to be filtered when `EXPORT_RAW=true` is used.

## Experimental phases

### Exploratory phase

Objective: identify the load range where each scenario starts to degrade.

Recommended configuration:

- `EXPERIMENT_PHASE=exploratory`;
- `EXPLORATORY_REPLICATIONS=3`;
- `RAMP_UP=1m`;
- `WARMUP=1m`;
- `DURATION=5m`;
- `COOLDOWN=30`;
- all initial progressive levels;
- additional levels with an approximate 50% increase until saturation.

This phase reduces computational cost and runtime because it avoids 30 replications on loads that will not be relevant to the final comparison.

The defaults in `scripts/run-k6-levels.sh` are shorter than the recommended configuration above so they can be used only for low-cost local smoke tests.

### Confirmatory phase

Objective: collect statistically relevant data at the levels selected after exploration.

Recommended configuration:

- `EXPERIMENT_PHASE=confirmatory`;
- `REPLICATIONS=30`;
- only the load levels chosen based on the exploratory phase.

The confirmatory set should include representative loads before degradation, near the degradation point, and, when useful, immediately after degradation begins.

## Degradation criteria

Stop progression when one or more signals appear clearly:

- substantial increase in p95 or p99 latency;
- throughput plateauing or dropping;
- material increase in error rate;
- saturated CPU, memory, disk, or network;
- service or orchestrator instability.

## Suggested procedure

1. Prepare the target scenario.
2. Confirm that `TARGET_URL` returns HTTP 200.
3. Confirm Prometheus targets.
4. Execute the exploratory phase with `scripts/run-k6-levels.sh`.
5. Select the relevant load levels.
6. Execute the confirmatory phase only at the selected levels.
7. Collect metadata with `scripts/collect-results.sh`.
8. Record external observations in a lab file.
9. Repeat runs at defined times to evaluate reproducibility.

## Minimum record per run

- UTC date and time.
- Scenario.
- Target host and load generator host.
- Where Prometheus and Grafana were running.
- Docker, Nginx, k6, and operating system versions.
- CPU, memory, and network configuration.
- Load level durations.
- Number of independent replications per level.
- Result files generated by k6.
- Observed abnormal events.

## Planned statistical analysis

For each combination of scenario and load level, calculate:

- mean and standard deviation of throughput;
- mean and standard deviation of mean latency;
- p95 and p99 latency;
- coefficient of variation of mean latency between replications;
- HTTP error rate;
- visual or statistical correlation with CPU, memory, network, and container metrics.

The article defines 30 independent runs per level and per scenario. For preliminary tests, the script accepts `REPLICATIONS=1`; for final collection, use `REPLICATIONS=30`.

The first automated summarization is in `analysis/summarize-k6-results.py`. It generates a simple CSV in `results/processed/` to support later analysis, without replacing the article's final statistical analysis.
